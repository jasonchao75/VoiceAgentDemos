# ASR 厂商接入调研技能 (ASR Survey)

## 使用场景
当需要调研接入一个新的 ASR（自动语音识别）厂商，或者相同厂商推出了新的模型、新的接口（如流式接口）需要进行技术调研与可行性验证时，使用此技能。
- 流式chunk size默认20ms, sample rate默认8000Hz，16bit 单声道PCM wav文件

## 执行步骤

### 1. 加载厂商专属知识库（显式强制调用）
在开始任何检索或编码之前，你**必须**优先使用 `skill` 工具尝试显式加载对应厂商的专属知识库技能。
- **技能名称规范**：`vendor-[厂商名小写]`（如 `vendor-deepgram`, `vendor-azure`）。
- **工具调用示例**：`[tool_call: skill for name="vendor-deepgram"]`
- **执行逻辑**：
  - 如果加载成功：仔细阅读加载出的厂商专属知识（鉴权陷阱、MCP工具使用、首包延迟痛点等），并将这些知识严格应用到后续的自测脚本中。
  - 如果加载失败（说明目前没有整理该厂商专属技能）：你才可以退而求其次，使用 `webfetch` 搜索该厂商官网或官方 Github 仓库查找最新 API 文档。重点寻找是否有官方提供的 MCP Server、GitHub 开源示例、以及相关开发者手册（API 文档）。

### 2. 过滤、去重与核查
- 筛选出真正有用的接口文档和鉴权说明，剔除过时或无关的营销信息。
- 核查厂商是否支持我们的特定需求（如电话场景的 8KHz / 16bit / 单声道 PCM 格式）。
- **结合第一步的专属知识**：如果已经加载了 `vendor-xxx` 技能，请重点核对其中的参数清单和已知坑点，避免遗漏。

### 3. 构建自测脚本与模拟流式测试
- **环境搭建**：向用户索要或使用用户已提供的 API Key，统一从 `.env` 环境变量中读取。
- **代码编写**：使用 Python 编写测试脚本。对于 Python，推荐使用 `asyncio` 和 `websockets` 库。
- **结合厂商知识**：严格按照第一步获取的专属知识设置 WebSocket Header、Query 参数和模型配置。对于有特殊签名算法的厂商（如腾讯云），优先寻找官方 SDK 替代手写 WebSocket。
- **双向并发协程**：流式测试必须并发处理发送和接收（例如使用 `asyncio.gather` 或 TaskGroup），避免串行读写导致连接断开。注意查看厂商是否需要发送 KeepAlive 心跳。
- **兜底音频策略**：如果当前工作目录下没有用于测试的 `.wav` 或 `.pcm` 文件，**绝不直接报错或停止**。请主动编写一段小逻辑（如利用 Python 的 `wave` 和 `math` 库），生成一段几秒钟、符合格式要求（如 8KHz/16bit 单声道）的正弦波或静音占位 PCM/WAV 文件，用于连通性测试。
- **日志要求**：在代码中加入详尽的日志打印（请求参数拼接过程、鉴权过程、首包延迟时间、完整识别结果等）。

### 4. 日志分析与脚本微调
- 运行自测脚本，查看输出日志。
- 发现报错或识别不准等问题时，优先排查鉴权 Header、并发逻辑以及网络代理问题，查阅专属知识微调脚本（流式chunk size默认20ms, sample rate默认8000Hz，16bit 单声道PCM wav文件）。

### 5. 输出最终验收脚本
- 最终确定一套稳定可用的脚本。
- 向用户交付该脚本，脚本必须可以直接运行、能打印清晰的调试与结果日志，以便用户进行产品级验收。

## 注意事项
- **交叉验证与兜底搜索 (防首因效应/锚定偏见)**：在下结论说“某个厂商不支持某项行业基础能力（如 WebSocket 双向流式）”之前，**必须**更换关键词（如 `<厂商名> 实时 WebSocket` 或 `<厂商名> 实时 SDK`）进行二次兜底搜索。**找不到证据不代表没有，必须用正向和反向关键词都查无结果，才能下结论**。
- **业务常识反推机制 (防信息茧房)**：如果搜索到的官方接口无法满足核心的业务诉求（例如发现 ASR 无法边发音频边接收识别结果），第一反应**绝不是去改变业务架构去迁就接口**，而是应当强烈怀疑“这不是业务的推荐接入方式”，并继续去寻找属于该业务（如实时交互 VoiceAgent）的专用 API（如 `OmniRealtimeConversation` 等双向接口）。
- **绝不硬编码密钥**：API Key 等敏感信息必须通过读取 `.env` 环境变量或命令行参数传入。
- **音频格式对齐**：必须确认测试的音频文件格式与厂商 API 要求的格式（如 8KHz, 16bit, 单通道 PCM）严格对应。
- **无测试音频时的处理**：如果本地没有合适的测试音频文件，不要直接报错或停止。请主动使用 Python（如 `wave` 和 `math` 库）生成一段符合格式要求（如 8KHz/16bit 单声道）的正弦波或静音占位文件用于连通性测试。
- **所有业务参数必须外置为Config（极其重要）**：在编写对接代码时，**绝对不能**将任何厂商支持的业务可调参数（如 `model`, `language`, `endpointing`, `utterance_end_ms`, `keyterm`, `smart_format` 等）硬编码在代码深处。这些参数必须**毫无遗漏地全部暴露**并放置在 `configs/vendor/{厂商名}/` 下的单独配置文件中（如 YAML 或 JSON）。测试的音频文件特征也必须和 config 中配置的参数（尤其是语言、采样率）严格一致。
- **真实连通性**：必须通过实际运行代码和查看日志来验证连通性，不能仅凭阅读文档给出结论。
- **防幻觉与官方文档备份（双轨制）**：在整理厂商专属技能文档（如 `vendor-xxx/xxx-docs.md`）时，必须采取“结构化摘要在顶部，原始 YAML/Markdown 源码兜底在底部”的严谨做法。绝不允许为了精简篇幅而删除官方原始文档内容。完整的原始规范是后续解决参数疑问或拦截大模型幻觉的终极事实锚点。
- **严格基于规范（防幻觉红线）**：在提取参数、编写代码或给出避坑建议时，**绝对禁止**凭借通用 ASR 知识进行臆断。你使用的所有 Query 或 Header 参数，**必须能在厂商的 OpenAPI / AsyncAPI 原文规范中找到精确的字段定义**。如果不确定，请相信规范原文，不要相信记忆。特别注意区分 REST API 独有的参数和 WebSocket 流式独有的参数，严禁混用。

## 输出格式
- **调研结论摘要**：简述厂商的支持情况（格式、延迟、特性）。
- **可运行的自测脚本**：以代码块形式提供，或写入到 `configs/vendor/{厂商名}/test_asr.py` 中。
- **运行日志示例**：附带一段成功的运行日志，证明接口连通。
- **验收说明**：告诉用户如何配置环境变量以及如何运行此脚本进行验收。

# Session Handoff (会话断点续传)

## 快速开始

在一个独立的工作单元结束时，使用以下格式输出进度快照：

```markdown
## 进度快照 (Progress Snapshot)

**生成时间**: <ISO 8601 时间戳>
**当前角色**: <Vendor-Researcher | Integration-Developer | Evaluation-Engineer>
**核心任务**: <一句话描述当前正在做的事>
**当前状态**: <进行中 In Progress | 阻塞 Blocked | 已完成 Completed>

### 已完成 (Completed)
- [x] <事项 1>
- [x] <事项 2>

### 待处理 (Pending)
- [ ] <事项 3>
- [ ] <事项 4>

### 关键决策 (Key Decisions)
| 决策项 | 原因说明 |
|--------|----------|
| <决策 1> | <为什么这么选> |

### 文件与 Git 状态 (Files & Git State)
| 文件路径 | 状态 | 备注 |
|----------|------|------|
| `<path>` | 已保存/未保存/已提交/未提交 | <备注> |

### 关键上下文 (仅限安全信息)
- 依赖环境变量: `<环境变量名>` (来源: `<配置文件路径或说明>`)
- 厂商特殊要求: <记录踩过的坑或特例>
- 阻塞原因 (若状态为 Blocked): <根本原因 + 复现步骤>

### 下一步建议 (Suggested Next Step)
**建议操作**: <供用户参考的下一步行动建议>
*(注意：这仅是建议。新会话开启后，Agent 将等待用户的明确指示后再行动)*
```

## 新会话开场白

在开启新对话时，将以下前缀与上面的快照一起粘贴进去：

```text
角色设定: <填入角色>
当前任务: <一句话描述任务>
强制约束: 请勿重复执行快照中标记为“已完成”的工作。
继续执行: 请读取并理解快照上下文，**随后等待我下达具体的下一步指令**，不要自行盲目开始。

<在此处粘贴进度快照>
```

## 何时触发断点续传

应在**自然的里程碑节点**主动触发，而不是等到上下文彻底乱掉：

- **Vendor-Researcher (调研员)**: 完成一家厂商的完整调研闭环后（文档获取完毕、验证脚本跑通、报告草稿写完）。
- **Integration-Developer (开发员)**: 完成一个适配器的代码编写并自测通过后。
- **Evaluation-Engineer (评测员)**: 完成一个评测维度后（例如，跑完了所有当前适配器的抗噪 WER 测试）。
- **角色切换**: 比如从调研员切换为开发员时，必须进行断点交接。
- **切勿等待**：不要等到 Agent 开始重复提问或无视红线规则了才想起来断点。

## 安全与底线规则

- **绝对禁止将 API Key 或 Token 明文写入快照**。只能记录环境变量的名称。
- **必须记录未保存的文件**。新会话只会从磁盘读取文件；如果有未保存的修改，必须先保存，否则状态会丢失。
- **必须记录未提交的 Git 变更**。如果下一个会话需要一个干净的工作区，必须先 commit 或 stash。
